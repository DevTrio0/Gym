# Quick Reference - ID Generation & User Registration

## System Guarantees

✅ **IDs are ALWAYS auto-generated by system**  
✅ **Users NEVER create their own IDs**  
✅ **Each ID is unique and system-assigned**  

---

## User Registration Flow

### Step 1: User Provides (In Registration Form)
```
- Name: "John Doe"
- Email: "john@example.com"
- Password: "securePassword123"
- Role: "coach" (user chooses from dropdown)
```

### Step 2: System Generates (Automatically)
```
- User ID:     "user_1707410800000_0.543..."  ← SYSTEM GENERATED
- Coach ID:    "coach_1707410800000_0.281..." ← SYSTEM GENERATED
- Status:      "pending"                       ← SYSTEM SET
- Created At:  "2026-02-08T10:30:00.000Z"    ← SYSTEM GENERATED
```

### Step 3: User Receives
```json
{
  "status": "success",
  "message": "Coach registered, pending approval",
  "userId": "user_1707410800000_0.543..."  ← User's auto-generated ID
}
```

- Password: "securePassword123"

### What System Returns
```json
{
  "status": "success",
  "message": "coach signed in",
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user": {
    "id": "user_1707410800000_0.543...",     ← User's system ID
    "name": "John Doe",
    "email": "john@example.com",
    "role": "coach"
  }
}
```

The **token contains the user's system-generated ID** - not a user-created one.

## Code Implementation

### ID Generation Function
```javascript
// In models/database.js
const getNextId = (prefix) => {
  return `${prefix}_${Date.now()}_${Math.random()}`;
};

// Usage:
const userId = getNextId('user');      // "user_1707410800000_0.543..."
const coachId = getNextId('coach');    // "coach_1707410800000_0.281..."
const clientId = getNextId('client');  // "client_1707410800000_0.756..."
```

### Registration (System generates ID)
```javascript
// In controllers/authController.js
const register = (req, res) => {
  const { name, email, password, role } = req.body;
  
  // System generates the ID - user doesn't provide it
  const userId = getNextId('user');  // ← SYSTEM GENERATES
  
  const newUser = {
    id: userId,              // ← System-generated
    name,                    // ← From user input
    email,                   // ← From user input
    password: hashedPassword,// ← From user input (then hashed)
    role,                    // ← From user input
    status: 'active',        // ← System set
    createdAt: new Date()    // ← System generated
  };
  
  mockDatabase.users.push(newUser);
  
  res.status(201).json({
    status: 'success',
    message: 'User registered',
    userId  // ← Return the system-generated ID
  });
};
```

### Sign In (System retrieves ID from token)
```javascript
// In controllers/authController.js
const signIn = (req, res) => {
  const { email, password } = req.body;
  const user = findUserByEmail(email);
  
  // Verify password
  const isValid = comparePassword(password, user.password);
  
  if (isValid) {
    // System generates token with user's system-generated ID
    const token = generateToken(user.id, user.role);
    
    res.status(200).json({
      status: 'success',
      token,
      user: {
        id: user.id,    // ← System-generated ID from registration
        name: user.name,
        email: user.email,
        role: user.role
      }
    });
  }
};
```

### Using ID in Requests (From JWT Token)
```javascript
// In all protected routes
const verifyBearerToken = (req, res, next) => {
  const token = getTokenFromHeader(req);
  const decoded = verifyToken(token);
  
  req.user = decoded;  // Contains: { userId, role, iat, exp }
  next();
};

// In controllers - accessing the user's system ID
const getClientProgress = (req, res) => {
  const coachId = req.user.userId;  // ← From JWT token (system-generated)
  
  const coach = mockDatabase.coaches.find(c => c.userId === coachId);
  // Find coach's clients and their progress
};
```

## What System Generates (Auto)
- ✅ User ID
- ✅ Role-specific ID (coach, client, admin)
- ✅ All resource IDs (diet, training, notes, payments)
- ✅ Reset tokens
- ✅ JWT tokens
- ✅ Timestamps

## What User Provides
- ✅ Name
- ✅ Email
- ✅ Password
- ✅ Role (selection)
- ❌ NOT the ID

---

## Important Notes

1. **IDs are immutable** - once created, cannot be changed
2. **IDs are unique** - random component ensures no duplicates
3. **IDs are traceable** - timestamp shows when created
4. **IDs are secure** - not sequential, hard to guess
5. **IDs are system-owned** - user has no control over them

This prevents:
- Unauthorized access by ID guessing
- User impersonation
- Manual ID manipulation
- Race conditions in ID generation
